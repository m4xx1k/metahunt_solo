The 202 + body change vs. legacy makes the curl smoke informative instead of returning empty 200. Spec verifies it calls `ingestAll` (not `ingestRemote`) exactly once with no args.
- RSS+Temporal port — T12 (`RssModule` wired into `AppModule`). Locked decisions applied: dropped `DatabaseModule.forRoot()` from RssModule's imports (already global at AppModule level), added `ExtractionModule` (provides `VACANCY_EXTRACTOR` for `RssExtractActivity`), `workflowsPath: resolve(__dirname, "workflows")`, webpack `ts-loader` hook for workflow bundling. - ETL `main.ts` env loading — Production scripts use `node --env-file-if-exists=../../.env`, but `nest start --watch` does not, so `pnpm start:dev` failed env validation locally. Added explicit `dotenv.config({ path: resolve(__dirname, "../../../.env") })` in `main.ts` (ts-resolved into `apps/etl/dist/`, so the path lands on the repo `.env`). `dotenv` does not override existing process env, so prod runtime is unaffected. Added `dotenv` to `apps/etl/package.json` runtime deps.

## 2026-05-03

- Daily RSS ingest now runs on a Temporal Schedule, not Nest's `@Cron`. `RssSchedulerService.onApplicationBootstrap` (re)installs schedule `rss-ingest-hourly` on every boot — calendar `{ minute: 0, hour: { start: 6, end: 22, step: RSS_INGEST_INTERVAL_HOURS } }` in `Europe/Kyiv`, overlap policy `SKIP`. The action starts a new `rssIngestAllWorkflow` which calls activity `listRemoteSourceIds` then fans out one `rssIngestWorkflow` child per source with `parentClosePolicy: ABANDON`. Existing `GET /rss` HTTP trigger and `ingestRemote/ingestAll` methods kept for manual runs.
- New env var `RSS_INGEST_INTERVAL_HOURS` (default 1, range 1..16) — change and restart the app to re-apply via the schedule handle's `update`.
- New activity `RssListSourcesActivity` + new workflow `rss-ingest-all.workflow.ts` (added to the `workflows/` barrel so the worker bundler picks them up). Suite: 13 suites / 48 tests, all green.
- Extraction is now **per-record best-effort** in `rssIngestWorkflow`: `Promise.allSettled` instead of `Promise.all`, so one bad vacancy (e.g. an LLM that mis-nests required fields) no longer fails the surrounding good records. Failure count is logged and recorded in `rss_ingests.error_message` as `extracted=X/Y (failures=N)`; `status` stays `completed` whenever fetch+parse succeeded.
- Activity retry policies normalized to `maximumAttempts: 3` with `5s/10s/20s` exponential backoff for `fetchAndStore`, `parseAndDedup`, `extractAndInsert`. `finalizeIngest` keeps 5 attempts (idempotent and critical for status correctness).
- BAML: `hasTestAssignment` and `hasReservation` switched from `bool` → `bool?`. Mini models occasionally lose scope on deeply-nested tail fields and omit them; nullable accepts that and unblocks the rest of the extraction. → regenerated `src/baml_client/`.
- Added `md/runbook/failure-recovery.md` — per-activity diagnosis + restore procedure for fetch / parse / extract failures, including how to pause the schedule during incidents.
- New endpoint `POST /rss/extract-missing?limit=N` — synchronous in-process backfill that re-runs `extractAndInsert` for `rss_records WHERE extracted_at IS NULL`, oldest first. Default 100, max 500. Recovers records that parse inserted but a prior failed run never extracted (dedup hides them from the schedule's next firing). Returns `{ attempted, succeeded, failed }`.
- Child workflow IDs are now human-readable: `rss-ingest-<code>-<YYYY-MM-DDTHH-MM-SSZ>` (e.g. `rss-ingest-djinni-2026-05-03T14-29-13Z`) instead of UUID-based. Activity `listRemoteSourceIds` → `listRemoteSources` returning `{ id, code }[]`; the workflow uses `code` for the ID and `workflowInfo().startTime` for the timestamp. Parent workflow ID stays `rss-ingest-all-<auto-suffix>` (the schedule already appends a date).
- Frontend imported into the monorepo as `@metahunt/web` (`apps/web/`) — clean copy from the standalone `metahunt-client` repo, no git history transfer. Renamed package, dropped `package-lock.json` in favor of the shared `pnpm-lock.yaml`. landing.pen moved to `apps/web/design/`. Backend Docker build is unaffected (Dockerfile copies selectively + `.dockerignore` adds `apps/web` for defense in depth). Vercel reconnect to point at this repo with `Root Directory = apps/web` is a one-time UI step covered by `md/runbook/vercel-reconnect.md`. → ADR-0005, [migration tracker](./migrations/frontend-migration.md).
